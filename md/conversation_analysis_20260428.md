# 系統異常分析報告：以 8455 (大拓-KY) 為例

## 1. 關於 8455 公司名稱問題

**修正與確認：** 您說得完全正確，8455 是「大拓-KY」，先前的回覆中我筆誤了。非常感謝您的糾正！

**深入分析未顯示名稱的原因：**
在 `check-company-type.js` 中，為了尋找上櫃公司名稱，程式呼叫了櫃買中心的 OpenAPI：
`https://www.tpex.org.tw/openapi/v1/tpex_mainboard_quotes`

這支 API 的全名是**「上櫃股票每日收盤行情(不含第一上櫃及外國企業)」**。
因為 8455 (大拓-KY) 屬於「外國企業 (KY股)」，所以它**被這支 API 排除在外**。
這導致 API 回傳成功，但裡面永遠找不到 8455 的資料。最後程式只好觸發保底機制，根據代碼特徵（4位數）自動顯示為「上櫃公司 8455」。

**解決思路：** 未來我們需要將抓取公司名稱的 API 換成包含所有上櫃公司的端點，例如 `mopsfin_t187ap03_O` (上櫃公司基本資料 API)，這樣 KY 股才不會被漏掉。

## 2. UI 互動優化需求：點擊色塊按鈕搜尋

**功能分析：**
目前介面的第 1 步是輸入代碼或名稱搜尋，而上櫃/上市/興櫃的色塊按鈕主要作為「全域過濾器」或自動檢測顯示。
如果您希望「點擊公司類型的色塊按鈕，然後輸入查詢條件搜尋」，這是一個很棒的 UX 優化方向：
1. 讓色塊按鈕變成一種「搜尋範圍鎖定」機制。
2. 點擊「上櫃公司」按鈕後，搜尋框會自動切換為只搜尋上櫃股票清單，提升搜尋速度與精準度。
3. 未來可透過前端 JavaScript 將點擊色塊的事件綁定到搜尋框的 `dataset` 過濾邏輯中。

## 3. 籌碼面數據為 0 的根本原因（TWSE vs TPEx）

**深入分析 API 來源：**
您指出的盲點非常精確：「如果是上櫃公司應該要找 tpex 的數據庫 API 資料，而不是只找 twse 的。」

目前的 `fetch-chips.js` 備援機制完全寫死了去呼叫台灣證券交易所（TWSE）的網址：
- 融資餘額：`www.twse.com.tw/rwd/zh/marginTrading/MI_MARGN`
- 三大法人：`www.twse.com.tw/rwd/zh/fund/TWT38U`

**問題所在：**
TWSE 只有**上市公司**的資料。當我們拿著 8455（上櫃）去問 TWSE 時，證交所會回傳找不到資料。
真正的上櫃公司籌碼資料，存放在**櫃買中心 (TPEx)** 的網站與 API 中。
- 上櫃融資融券餘額 API 應該使用 TPEx 的專屬端點（如 `margin_bal_result`）。
- 上櫃三大法人買賣超 API 應該使用 TPEx 的端點（如 `3itrade_hedge_result`）。

因為目前 `fetch-chips.js` 中缺乏針對上櫃公司 (OTC) 切換到 TPEx API 的路由判斷（Routing Logic），所以遇到上櫃公司且 FinMind 失敗時，籌碼數據就會直接呈現為 0 或是無法取得。

## 總結
1. **名稱缺失**：因為當前的 TPEx OpenAPI 排除掉了 KY 股 (外國企業)。
2. **籌碼為零**：因為遇到上櫃公司時，後端備援沒有切換到櫃買中心 (TPEx) 專屬的 API，仍錯誤地向證交所 (TWSE) 查詢。
